## מה ולמה?
ריפקטורינג הוא התהליך של שיפור הקוד שלך כך שיהיה נקי יותר, נגיש יותר לתחזוקה, ללא שינוי בהתנהגותו החיצונית. מתכנתים מבצעים ריפקטורינג כדי לשפר את הקריאות, להפחית את המורכבות, וכדי להפוך את מאגר הקוד ליותר נתיר לעדכונים או הוספות פיצ'רים בעתיד.

## איך לעשות?
נניח שיש לך קטע קוד בו אתה ביצעת חישובים או מניפולציות מחרוזות חוזרות על עצמן במספר פונקציות. זהו מוקד מושלם לריפקטורינג. הנה דוגמה לפני ואחרי באמצעות Gleam, שמדגישה חוזק בבטיחות טיפוסים ובלתי שינוי:

```gleam
// לפני ריפקטורינג
pub fn calculate_area(width: Int, height: Int) -> Int {
  width * height
}

pub fn print_area(width: Int, height: Int) {
  let area = calculate_area(width, height)
  io.println("The area is \(area)")
}

// אחרי ריפקטורינג
pub fn calculate_area(width: Int, height: Int) -> Int {
  width * height
}

pub fn print_area(area: Int) {
  io.println("The area is \(area)")
}

// בחלק אחר של הקוד שלך, תקרא ל-print_area כך:
print_area(calculate_area(10, 20))
```

פלט לדוגמה:
```
The area is 200
```

על ידי ריפקטורינג, עשינו את `print_area` יותר ממוקד בדפוס בלבד, בעוד שהחישוב מתבצע במקום אחר, ובכך הקוד נהיה יותר מודולרי וקל יותר לשימוש חוזר או לבדיקה.

## טבילה עמוקה
ריפקטורינג, כמושג, קיים כל עוד הפרוגרמינג עצמו - לבקר ולנקות קוד הוא חלק מניקיון טוב. הפורמליזציה המודרנית של ריפקטורינג, יחד עם רבות מהטכניקות והדפוסים המשמשים היום, יכולים להיות מוחזרים לספרו המהותי של מרטין פאולר "Refactoring: Improving the Design of Existing Code" שפורסם ב-1999.

באקוסיסטם של Gleam, ישנם שיקולים ספציפיים לריפקטורינג. אחד הדברים החשובים ביותר הוא הבדיקה החזקה של טיפוסי הנתונים בזמן ההדרה, שיכולה לעזור לתפוס טעויות מוקדם כשאתה מעביר דברים ממקום למקום. התאמת התבניות ותכונות הבלתי-שינוי של Gleam יכולות גם להדריך אותך לכתוב קוד יותר ברור וצמצום - אחד המטרות העיקריות של ריפקטורינג.

חלופות לריפקטורינג עשויות לכלול כתיבת הקוד מחדש מאפס או תיקון קוד עם פתרונות זמניים. עם זאת, ריפקטורינג, בדרך כלל, הוא הגישה הבטוחה והיעילה ביותר לשיפור קוד קיים ללא הכנסת באגים חדשים, מכיוון שהוא כולל טרנספורמציות שמרניות, מדוקדקות ושומרות התנהגות.
