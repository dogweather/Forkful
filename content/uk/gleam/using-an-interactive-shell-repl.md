## Що та Чому?

REPL, що розшифровується як Read-Eval-Print Loop (Цикл Читання-Виконання-Виведення), є інструментом програмування для інтерактивного виконання коду та миттєвого перегляду результатів. Програмісти використовують його для експериментів, налагодження або вивчення нової мови на льоту, як от Gleam.

## Як це зробити:

Наразі в стандартній дистрибуції Gleam немає REPL. Однак, ви можете експериментувати з кодом Gleam, використовуючи існуючу оболонку Erlang, оскільки Gleam компілюється в байткод Erlang. Ось як це зробити:

1. Компілюйте ваш код Gleam в Erlang.
```plaintext
gleam build
```

2. Запустіть оболонку Erlang.
```plaintext
erl -pa ebin
```

3. Викликайте свої функції Gleam (припустимо, що у вас є модуль `my_mod` та функція `my_fun`).
```erlang
my_mod:my_fun().
```

Ви маєте бачити вивід вашої функції в оболонці.

## Поглиблено

REPL випромінює динамічний та дослідницький дух багатьох функціональних мов програмування, йдучи корінням до REPL у LISP 1960-х. В порівнянні, інші системи, як-от `ipython` у Python або `irb` у Ruby, пропонують подібний досвід для своїх спільнот.

Хоча нативного REPL для Gleam поки немає, використання оболонки Erlang залишається хитромудрим обхідним шляхом. Здатності оболонки Erlang походять з BEAM VM, віртуальної машини, що живить екосистему Erlang, до якої належать Elixir, LFE та Gleam.

Альтернативами REPL у екосистемі Gleam можуть бути написання тестових випадків або використання онлайн компіляторів та ігрових майданчиків для коду, що підтримують Gleam, для перевірки фрагментів коду поза повноцінною налаштованою середовищем проекту.

Впровадження присвяченого REPL для Gleam стикається із викликами, зокрема, через скомпільовану природу Gleam та середовище виконання Erlang, де нормою є гаряча заміна коду. Будь-який майбутній REPL для Gleam потребуватиме узгодження статичної типізації мови із динамічним середовищем виконання, якого очікує REPL.
