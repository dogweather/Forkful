## क्या और क्यों?
एरर्स को संभालना इस बारे में है कि आपके कोड में चीजें अनपेक्षित रूप से गलत होने पर उन स्थितियों का कुशलतापूर्वक प्रबंधन कैसे किया जाए। प्रोग्रामर इसे इसलिए करते हैं क्योंकि यह ऐप्लिकेशन्स को मजबूत और यूजर-फ्रेंडली बनाता है, यहां तक कि अप्रत्याशित परिस्थितियों का सामना करते समय भी।

## कैसे करें:
Gleam में, आप अक्सर एरर हैंडलिंग के लिए `Result` प्रकार (type) का उपयोग करेंगे। यह एक enum है जिसमें दो वेरिएंट होते हैं: `Ok` (सफलता के लिए) और `Error` (विफलता के लिए)। यहां एक साधारण उदाहरण है:

```Gleam
pub fn might_fail(break_it: Bool) -> Result(Int, String) {
  if break_it {
    Error("Oops! It broke.".to_string())
  } else {
    Ok(42)
  }
}

pub fn main() {
  let result = might_fail(False)
  case result {
    Ok(value) => value
    Error(message) => {
      io.println(message)
      0
    } 
  }
}
```

यदि आप `main` को `might_fail(False)` के साथ चलाते हैं, तो यह `42` लौटाएगा। यदि आप `True` पास करते हैं, तो यह "Oops! It broke." प्रिंट करता है और `0` लौटाता है।

## गहराई से समझें
Gleam की एरर हैंडलिंग की प्रक्रिया इसके Erlang मूल से प्रभावित होती है। ऐतिहासिक रूप से, Erlang का एक "इसे गिरने दो" का दर्शन है, जो प्रक्रिया विफलताओं को प्रबंधित करने के लिए सुपरविजन ट्रीज पर निर्भर करता है। हालांकि, जब आप Gleam कोड लिख रहे होते हैं जो किसी सुपरवाइज किए जाने वाले प्रोसेस के भीतर नहीं है, जैसे कि एक लाइब्रेरी फंक्शन के भीतर, आप एरर्स को स्पष्ट रूप से संभालना चाहेंगे।

`Result` का उपयोग करने के विकल्पों में `Option` प्रकार का उपयोग करना शामिल है, जहां कुछ `None` (कुछ भी नहीं) या `Some` (कुछ) हो सकता है, परन्तु ये एरर जानकारी नहीं रखते। प्रक्रिया सीमाओं के पार एरर्स को संकेतित करने के लिए, आप Erlang के मैसेज-पासिंग तंत्र का उपयोग कर सकते हैं।

Gleam की एरर हैंडलिंग एक फंक्शनल प्रोग्रामिंग शैली को दर्शाती है, जहां साइड-इफेक्ट्स (जैसे कि एरर्स) को प्रकारों और पैटर्न-मैचिंग के साथ प्रबंधित किया जाता है, जो एरर प्रबंधन में स्पष्टता और अनुमानितता प्रदान करता है।
